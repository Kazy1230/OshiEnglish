1. **テーブルリネームの順序バグ**：`Base.metadata.create_all(bind=engine)`が`_rename_table()`より前に走っていたため、

   テーブルリネームを今後追加する場合も、必ずcreate_all()より前に書くこと。
2. **FK制約があるカラム/テーブルのDROPが失敗する**：`DROP TABLE articles`が`exercise_submissions`からのFK参照で失敗、
   `DROP COLUMN customers.assigned_admin_id`が`customers_ibfk_1`制約で失敗。
   → 修正：レガシーテーブルのDROPループを`SET FOREIGN_KEY_CHECKS=0/1`で囲み、
   `_drop_column()`に`_drop_foreign_keys_on_column()`（対象カラムのFK制約を先に自動DROPする）を追加。
3. 軽微: drop対象テーブル名のtypo（`message_feedbacks`→正しくは`message_feedback`）を修正。

また、**ローカルdocker-compose.ymlのDB認証情報が`.env`の実際の値と食い違っていた**問題も発見・修正：
`db`/`backend`サービスがハードコードされた弱い認証情報（`root_pass`/`yt_pass`）を使っていたが、
MySQLボリュームは`.env`の強力なパスワードで初期化されていたため接続できなかった。
→ 修正：`docker-compose.yml`の`db`/`backend`サービスを`env_file: .env`方式に変更（`docker-compose.prod.yml`と同じパターン）。

**確認結果**：`docker compose ps`で5コンテナ全てhealthy。`/health`, `/api/courses`, `/api/creators/`, `http://localhost/`が200を返すことを確認。
マイグレーションは再起動しても冪等（2回目はno-op、エラーなし）であることも確認済み。

**⚠️ 設計書からの重要な逸脱（必ず把握すること）**：
設計書は「90日を1回のAPI呼び出しで一括生成（30〜60秒）」を想定しているが、これは**実装しなかった**。理由：
1. 90日分を1回のLLM呼び出しで生成すると出力トークン量が膨大になり、JSON崩れ（出力途中での切れ）のリスクが高い。実機検証で実際に途中で切れることを確認した
2. そのため**週単位（13回のAI呼び出し、1〜12週=7日分/回・13週=6日分/回）に分割**する設計に変更した
3. これにより全体の生成時間は**実測で約12〜15分**かかる（設計書の想定「30〜60秒」を大幅に超える）。13週×1回あたり50〜70秒程度
4. 1リクエストで13回ぶん同期的に待つとnginx/ブラウザ側がタイムアウトするため、**FastAPIの`BackgroundTasks`で非同期実行**し、フロントエンドは`GET /courses/{id}/generation-status`を4秒間隔でポーリングしてプログレスバーを表示する設計にした（`days_generation_status`: idle/generating/completed/failed）
5. 週ごとに`db.commit()`するため、生成中でも`GET /courses/{id}/days`で途中経過を取得できる

この生成時間の長さ（12〜15分）はプロダクトのUX上の課題になり得る。Phase7-8以降でAnthropic Batch APIへの移行を検討する余地がある（ただしBatch APIは非同期でレスポンスまで時間がかかるため、コース作成フローには不向きな可能性がある。要検討）。
