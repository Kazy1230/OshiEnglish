# ManaVillage コミュニケーション設計 詳細仕様

> ステータス: 現行コード準拠版
> 関連ドキュメント: ManaVillage 要件定義書 v1.1 / Day1初回診断フロー詳細仕様

---

## 1. 概要

学習者からの相談・質問への対応は、コースのティアによって異なる。
どちらのティアでも質問は蓄積・分析され、クリエイターのコンテンツ改善に活用される。

回答の口調・指導スタイルは、クリエイターごとに1つだけ存在する「人格プロファイル」
（AIインタビューで抽出）に基づいて生成される。1クリエイター=1人格が原則であり、
チャット回答・朝夕の声かけ通知・督促リマインド・週次/月次レビュー・コース紹介文など、
学習者と接するすべてのAIメッセージがこの人格プロファイルを参照する。

## 2. コースティア定義

| ティア | 名称 | 回答者 | 価格帯 |
|---|---|---|---|
| Tier A | AI伴走コース | AIのみ | 980〜1,980円/月 |
| Tier B | プレミアム伴走コース | 講師が1日1回直接回答 | 2,980〜5,000円/月 |

## 3. 学習者からの相談の種類

| 種別 | 例 |
|---|---|
| ① 感情・モチベーション系 | 「単語のモチベがなくなっちゃって...」「続けられるか不安」 |
| ② 学習内容の質問系 | 「仮定法ってどう覚えればいい?」「Part 3が全然取れない」 |
| ③ 状況報告・雑談系 | 「今日30分やりました」「昨日できなかった...」 |

## 4. Tier A: AI伴走コースの回答フロー

```
学習者が質問を送信
  ↓
AIが質問を自動タグ付け(カテゴリ分類)
  ↓
紐付けコンテンツの検索
  ├── 該当コンテンツあり → コンテンツ+クリエイター口調のメッセージを返信
  └── 該当コンテンツなし → 人格プロファイルのみで回答生成
  ↓
質問ログを蓄積(分析ダッシュボードに反映)
```

### 回答生成の優先順位

1. **クリエイターが紐付けたコンテンツがある場合**
   - そのコンテンツ(動画URL・記事等)を提示
   - 人格プロファイルを適用した紹介メッセージを添える

   ```
   返信例:
   「仮定法、悩むよね。実はこれについて動画を作ったんだ。
   まずこれを見てみて！→ [仮定法マスター動画]
   見てわからなかったらまた聞いてね。」
   ```

2. **紐付けコンテンツがない場合**
   - 人格プロファイル + 学習者の診断結果をもとにAIが回答
   - 種別ごとに回答スタイルを変える

   | 種別 (message_type) | 回答スタイル |
   |---|---|
   | emotion（感情・モチベーション系） | まず共感→原因の整理→小さな行動提案 |
   | content（学習内容の質問系） | 結論→理由→具体例→次のアクション |
   | report（状況報告・雑談系） | 労い→翌日への橋渡し |

   回答生成時は、人格プロファイル部分（コース内で不変）とこの回答スタイル部分（種別ごとに可変）を
   別々のcontent blockに分け、人格プロファイル側にのみ`cache_control`（Anthropic Prompt Caching）を
   指定してAPIコストを削減している。

### モデルの使い分け

学習内容の質問・感情系の相談（emotion / content）は継続率に直結するため上位モデルで生成する。
状況報告・雑談系（report）は定型応答で十分なため軽量モデルを使うが、
解約・クレーム等の機微な語句を含む場合や300文字を超える長文の場合は上位モデルにエスカレーションする。

### フラストレーション検知（Tier Bアップグレード提案）

Tier Aの学習者が直近7日間に同一カテゴリ（承認済みのもののみ）へ3回以上質問している場合、
回答と同時にフラストレーション信号（トピック名・件数）を返し、Tier Bへのアップグレードを提案する
動線に利用する（事業検証ポイント①、しきい値はMVP後に実データで調整予定）。

## 5. Tier B: プレミアム伴走コースの回答フロー

```
学習者が質問を送信
  ↓
AIが質問を自動タグ付け(カテゴリ分類)
  ↓
AIが下書き回答を生成(人格プロファイル適用、常に上位モデルで生成し品質を優先)
  ↓
講師にメール通知
  ↓
講師が管理画面で下書きを確認
  ├── 未指定で送信 → AI下書きをそのまま承認・送信
  └── 編集して送信 → 編集内容を送信
```

24時間以上未対応の質問は`is_overdue=true`としてAPIが返し、一覧の先頭に並べ替えて表示する
（ログイン直後に最優先で目に入るようにする）。24時間超過件数はダッシュボードの優先通知バッジ用に
別エンドポイントで取得できる。学習者側に「講師が忙しい」等の自動表示メッセージは出さない
（現時点では未実装）。

### 講師向け管理画面(回答UI)

| 要素 | 内容 |
|---|---|
| 未回答一覧 | 質問内容・カテゴリ・受信時刻・経過時間・24時間超過フラグ |
| AI下書き | 人格プロファイルで生成した回答案(`ai_draft`) |
| 回答送信 | 下書きをそのまま送信、または本文を指定して編集送信(1件ずつ) |

### 1日1回の回数制限について

- 学習者1人につき1日1回まで講師への質問が届く(`status=pending_instructor`の本日分の有無で判定)
- 2回目以降の質問はAIが自動回答(Tier Aと同じフロー、`status=answered_by_ai`)

## 6. 質問分析ダッシュボード(クリエイター向け)

両ティア共通で、全質問を蓄積・分析してクリエイターに提供する。

### 表示内容

#### ① 質問カテゴリ別ランキング

```
【今週の質問Top5】
1位: 仮定法 (12件)
2位: モチベーション維持 (8件)
3位: Part 3 リスニング (7件)
4位: 単語の覚え方 (5件)
5位: 学習時間の確保 (4件)
```

#### ② カテゴリ別詳細

各カテゴリをクリックすると実際の質問文一覧を確認できる。

```
「仮定法」カテゴリの質問(12件)
- 「仮定法過去完了がいつも混乱します」
- 「If節の使い分けがわからない」
- 「仮定法ってどう覚えればいいですか?」
...
```

#### ③ コンテンツ紐付けボタン

質問カテゴリの横に「コンテンツを紐付ける」ボタンを設置。
クリックすると動画URL・記事URLを登録できる。

## 7. コンテンツ×質問の紐付けループ

ManaVillageの差別化を生む核心的な仕組み。

```
① 学習者が「仮定法」について多く質問する
  ↓
② ダッシュボードに「仮定法(12件)」と表示される
  ↓
③ クリエイターが仮定法の動画を作成してYouTube等に投稿
  ↓
④ 管理画面で「仮定法」カテゴリに動画URLを紐付ける
  ↓
⑤ 以降、「仮定法」に分類される質問が来たら
    AIが自動でその動画を含めて回答する
  ↓
⑥ 新しい質問が来るたびにループが繰り返される
   → クリエイターの知識が蓄積されてAIの回答品質が上がっていく
```

### 紐付けデータ構造

```
question_categories
  - id
  - creator_id
  - name          (例: "仮定法")
  - keywords      (JSON: 現状未使用。将来のキーワードマッチング拡張用に予約)
  - status        (pending / approved / rejected)
  - created_at

category_contents
  - id
  - category_id
  - content_type  (video / article / pdf)
  - title
  - url
  - created_at
```

## 8. 質問の自動タグ付け仕様

### 分類方法

AIが質問文を読み、以下の2項目を1回のLLM呼び出しでJUDGEする(軽量モデルで分類)。

| 項目 | 内容 |
|---|---|
| category_name | 学習コンテンツ軸での分類名(例: 仮定法・リスニングPart3・単語の覚え方・モチベーション維持・学習時間の確保 等) |
| message_type | emotion / content / report のいずれか(回答スタイルの選択に使う。3章参照) |

category_nameは、既存カテゴリ一覧をプロンプトに渡し「当てはまるものがあれば必ずそのまま使う」よう
指示することで表記揺れを防いでいる。`keywords`列でのキーワード照合は現状実装されていない。

### タグ付けロジック(承認制)

1. AIが提案したcategory_nameが既存カテゴリ名と一致すれば、その既存カテゴリに分類する
2. 一致しない場合は新規カテゴリを`status=pending`で自動作成する
3. `pending`のカテゴリは、コンテンツ紐付け・質問分析ダッシュボード・フラストレーション検知の対象には
   ならない。クリエイターが管理画面で「承認(approved)」「却下(rejected)」を選ぶまで保留される
4. 却下(rejected)後も、既に紐付いた質問への分類自体は残るが、分析・検知対象からは外れる

## 10. 人格プロファイル（AIインタビューによる人格収集）

クリエイターの口調・指導哲学・判断基準を、AIインタビューを通じて抽出し、`personality_profiles`に
JSONとして保存する。1クリエイターにつき1レコード（1人格=1キャラクター）。

### インタビューの問い方

「あなたの指導哲学は？」のような抽象的な質問では建前の回答になりやすいため、学習者本人のセリフを
そのままクリエイターに投げかけるロールプレイ形式を採用している。固定5問（例:
「○○先生、最近全然リスニングが伸びなくて…正直心が折れそうです。」等）を順に出し、
回答が短い・抽象的・臨場感がないとAIが判定した場合は、同じ学習者がさらに食い下がるセリフ形式の
深掘り質問（最大3問まで）を追加する。深掘りが不要と判定されるか上限に達すると次の固定質問に進む。

進行状態（現在の質問インデックス・深掘り回数・回答履歴）は`interview_sessions`に保存され、
ブラウザを閉じても途中から再開できる。

### Step0: 指導スタイル・性別プリセット

インタビュー開始時に、指導スタイルのプリセット（共感型/指導型/激励型/厳格型）とキャラクターの性別
（男性/女性/中性的）を任意で選択できる。これらはインタビュー回答が薄い項目を補うヒントとして
プロファイル生成プロンプトに渡されるのみで、回答内容そのものを優先する。

### 人格プロファイルのJSON構造

インタビュー全文（学習者のセリフとクリエイターの返答のペア、深掘り含む）から、AIが以下の構造で
プロファイルを生成する。

```json
{
  "communication": {
    "tone": "口調の特徴",
    "first_person": "一人称",
    "sentence_ending": "語尾の特徴",
    "catchphrase": "よく使う口癖・決め台詞"
  },
  "coaching_style": {
    "strictness": "厳しさの度合いと特徴",
    "encouragement": "励まし方の特徴",
    "feedback_method": "フィードバックの方法"
  },
  "learning_philosophy": {
    "core_value": "最も重視する考え方",
    "priority": "優先順位の付け方",
    "judgment_criteria": "判断基準"
  },
  "thinking_style": {
    "analogy_tendency": "例え話の癖",
    "explanation_method": "説明方法の特徴",
    "problem_solving": "問題解決アプローチ"
  }
}
```

`sample_reply`（学習者が「最近やる気が出なくて、続けられるか不安です…」と相談した場合にこの
クリエイターが実際に返すであろう一言）も同時に生成されるが、`profile`本体とは別に
`personality_profiles.sample_reply`列へ1回だけ生成・保存され、クリエイター紹介ページの
サンプルチャット表示に使われる（都度生成はしない）。

回答に書かれていない項目があっても、文体・語尾・話の運び方から推測してAIが全項目を埋める。
クリエイターは生成後にプロファイルをAPI経由で手動修正できる。

### コーチング/スキルタグの表示

クリエイター紹介ページでは、`profile`から追加のLLM呼び出しなしで2種類の説明文を組み立てて表示する。

| タグ | 組み立て元 | 内容 |
|---|---|---|
| coaching_tags（スタンス） | `coaching_style.strictness` + `encouragement` + `feedback_method` | 指導の性格・方針 |
| skill_tags（スキル・特徴） | `learning_philosophy.core_value` + `thinking_style.explanation_method` + `analogy_tendency` | 技術・専門性側の特徴 |

いずれも3項目を「。」で連結し、要素数1の配列として返す（フロントエンドの型を変えずに表示は1項目に
統合している）。

## 11. 朝夕の声かけ通知・督促リマインド

人格プロファイルを反映したAIメッセージを、バッチ処理で都度生成して送信する（プリ生成はしない）。

### 朝/夜の声かけ（Layer3）

学習者が設定した朝・夜の通知時刻（JST基準）になると、その日のタスク（Layer2）と直近3日分の
要約サマリーを踏まえて、その場でメッセージを生成し`notifications`に保存、メールでも送信する。
休養日（is_rest_day）の場合は送信しない。同じ日・同じスロット（morning/evening）への二重送信は
防止される。Day番号はDay1診断完了日（`learner_profiles.created_at`）からの経過日数で算出し、
最大30日でキャップする。

### 未活動リマインド（3段階）

学習者が最後にチャットを開いた（質問を送信した）日時、またはDay1診断完了日時を起点に、
未活動日数（days_inactive）が1日以上になるとリマインドメールを送る（1日1コースあたり1通まで）。
トーンは未活動日数に応じて3段階に変化する。

| 段階 | トーン |
|---|---|
| 1日 | 通常の声かけ。今日のタスクを確認しようと軽く促す |
| 2日 | 促進トーン。昨日できなかった分を一緒に取り戻そうと前向きに励ます |
| 3日以上 | 感情に寄り添うトーン。プレッシャーをかけず温かく呼びかける |

未活動が4日以上続いた場合は、Tier B講師に対して「学習者がN日間チャットを開いていません」という
メールを送り、直接連絡を促す。完了済み日次ログが30件に達した学習者（コース完了済み）は対象外。

## 12. 週次・月次レビュー

学習者は自分の週次・月次レビュー一覧を取得できる（`learner_reviews`、新しい順）。
各レビューは`review_type`（週次/月次）・`period_number`（第何週/第何月）・`content`を持つ。
