# 変更v1: DBスキーマ移行(Step1) - ロール管理・講師プロフィール導入・クレジット制廃止

> 関連ドキュメント: ManaVillage 要件定義書 / 基本設計書 / 詳細設計書 / 修正点.md
> 対応範囲: 修正点.md「4. 開発の着手順序の提案」のStep1(DBスキーマ移行)

---

## 1. 背景

ManaVillageマーケットプレイス化にあたり、既存のOshi Englishのデータベース・バックエンドコードを
「学習者・講師・管理者」のロール構造、講師ごとのキャラクター所有、コース単位の購入モデルに対応させる
ための最初のステップ。本プロジェクトにAlembicは実質導入されておらず、`backend/app/main.py`内の
`_ensure_column`/`_drop_column`(簡易マイグレーション①〜㉛)が実質的な既存マイグレーション機構のため、
これに倣い「簡易マイグレーション㉜」として追加した。

---

## 2. 新規テーブル

モデルファイルを追加し、`Base.metadata.create_all()`により起動時に自動作成される。

| テーブル | モデルファイル | 概要 |
|---|---|---|
| `instructor_profiles` | `backend/app/models/instructor_profile.py` | 講師プロフィール(bio・SNSリンク・審査status)。`customers.id`に1:1で紐づく |
| `courses` | `backend/app/models/course.py` | 購入単位のコース(キャラクター・価格・カテゴリ・公開状態) |
| `lessons` | `backend/app/models/lesson.py` | コース内の個別レッスン(テキスト/動画) |
| `purchases` | `backend/app/models/purchase.py` | コース単位の購入履歴(Stripe決済) |
| `lesson_progress` | `backend/app/models/lesson_progress.py` | レッスン単位の学習進捗 |
| `favorites` | `backend/app/models/favorite.py` | お気に入り講師 |
| `notifications` | `backend/app/models/notification.py` | 通知 |

`backend/app/models/__init__.py`に全モデルを登録済み。

---

## 3. 既存テーブルの変更

`backend/app/main.py`の「簡易マイグレーション㉜」ブロックで実施(起動時に自動・冪等に実行)。

### customers
- `role` (VARCHAR, learner/instructor/admin) を追加。既存`is_admin=1`のレコードは`role='admin'`にバックフィル
- `is_admin` カラムを削除
- `credit_balance` カラムを削除

### characters
- `instructor_id` (FK → `instructor_profiles.id`) を追加
- `is_preset` カラムを削除

### articles
- `unlock_cost` カラムを削除

### テーブル削除
- `credit_transactions`
- `credit_settings`

> **テーブル名についての判断**: 基本設計書では`users`テーブルとなっているが、既存実装では`customers`
> テーブル(`Customer`モデル)が約20ファイルから参照されているため、**今回は改名せず`customers`のまま**
> ロールカラムを追加した。`users`への改名は影響範囲が大きいため別タスクとする。

---

## 4. データ移行: 既存キャラクターの講師プロフィールへの割り当て

`backend/app/core/instructor_migration.py`に共通関数`migrate_legacy_characters_to_instructor()`を実装。

- `instructor_id`が未設定の既存キャラクター(白河雪菜・蒼井零・ドラえもん風・鬼島先輩・Chloe・Frederick等)
  を全件取得
- `manavillage_official`という講師アカウント(`role='instructor'`)が存在しなければ作成し、対応する
  `instructor_profiles`(`status='active'`)を用意
- 対象キャラクター全てに`instructor_id`を割り当てる(Phase 1は自社運営のみのため1つの講師に集約)

実行経路は2つ:
1. `main.py`起動時に自動実行(冪等。既に割り当て済みなら何もしない)
2. 手動実行用スクリプト: `backend/scripts/migrate_legacy_characters_to_instructor.py`
   (`docker compose exec backend python scripts/migrate_legacy_characters_to_instructor.py`)

---

## 5. クレジット制決済の廃止に伴うコード変更

`credit_balance`/`CreditTransaction`/`unlock_cost`をDBから削除したことに伴い、依存していた約15ファイルの
コードを「クラッシュしない最小限の無効化」まで対応した(Step3でcourses/purchasesによる正式な購入フローに
置き換える前提のスタブ)。

| ファイル | 変更内容 |
|---|---|
| `backend/app/core/credits.py` | `grant_credits`/`consume_credits`を何もしないスタブに変更。`get_credit_settings`は削除 |
| `backend/app/core/template_articles.py` | `unlock_cost`設定・`get_credit_settings`呼び出しを削除 |
| `backend/app/routers/auth.py` | ログインボーナスのクレジット付与を削除(親密度ポイント付与のみ残す)。`is_admin`→`role`に置換 |
| `backend/app/routers/customers.py` | `is_admin`→`role`に置換。`CreditTransaction`削除処理・レスポンスの`credit_balance`を削除 |
| `backend/app/routers/messages.py` | DM送信・記事リクエストのクレジット消費を削除。`/admin/{id}/credits/adjust`エンドポイントを削除。`is_admin`→`role`に置換 |
| `backend/app/routers/articles.py` | 記事の開封課金ロジックを撤去(常に`locked=False`)。`unlock_cost`関連スキーマ・ロジックを削除 |
| `backend/app/routers/payments.py` | クレジット購入エンドポイント(`/credits/checkout-session`)・Webhookハンドラを削除。サインアップ時のクレジット付与を削除。`is_preset`参照をキャラクター名リストでの暫定判定に置換 |
| `backend/app/routers/characters.py` | スキーマから`is_preset`を削除し`instructor_id`を追加 |
| `backend/app/routers/rewards.py` / `backend/app/core/rewards.py` | `character.is_preset`参照を削除(暫定的に全キャラ・上限の大きい方を適用) |
| `backend/app/core/security.py` | `get_current_admin`の判定を`is_admin`→`role == "admin"`に変更 |
| `backend/seed.py` | `is_admin=True`→`role="admin"`、`is_preset=True`指定を削除 |

## 8. 残課題(フロントエンド)

`is_admin` / `is_preset` / `credit_balance` / `unlock_cost`を参照しているフロントエンドファイルが
18件ある(`frontend/app/admin/tabs/*.tsx`、`frontend/app/credits/page.tsx`、`frontend/app/shelf/page.tsx`、
`frontend/app/chat/page.tsx`、`frontend/app/login/page.tsx`等)。今回のスコープはバックエンドDBスキーマ
移行のみのため未対応。**このままフロントエンドを起動すると、上記フィールドを参照する画面でAPIレスポンス
不整合が発生する。** 次のステップとしてフロントエンドの追従修正が必要。
