## 1. ①コース・レッスンのCRUD API(バックエンド)

新規ルーター [backend/app/routers/courses.py](../backend/app/routers/courses.py) を追加し、`main.py` に登録した。

### 1.1 主要なヘルパー

- `_get_owned_course(db, course_id, current_user)`: コース存在チェック(404)、`role != admin` の場合は
  `Course.character.instructor_id → InstructorProfile.id → InstructorProfile.user_id == current_user.id`
  の所有権チェック(403)
- `_is_purchased(db, user_id, course_id)`: `Purchase.status == "succeeded"` の有無で判定
- `_serialize_lesson(lesson, unlocked)`: `unlocked or lesson.is_preview` でない限り `body`/`youtube_url` を `null` で返す

### 1.2 エンドポイント一覧

| メソッド | パス | 認証 | 概要 |
|---|---|---|---|
| GET | `/courses` | 任意 | 公開済み(`status='published'`)コース一覧。`category`クエリで絞り込み可 |
| GET | `/instructors/{instructor_id}/courses` | 任意 | 指定講師の公開済みコース一覧 |
| GET | `/courses/{course_id}` | 任意(optional auth) | コース詳細+レッスン一覧。未購入かつ`is_preview=false`のレッスンは`body`/`youtube_url`を`null`化。非公開コースは所有者/adminのみ閲覧可 |
| POST | `/courses` | instructor/admin | コース新規作成(所有キャラクターのみ指定可) |
| PUT | `/courses/{course_id}` | 所有者/admin | コース更新。`status`を`published`に変更する際、レッスン1件以上を要求し、講師への`Favorite`登録者全員に`Notification(type='new_course')`を生成 |
| POST | `/courses/{course_id}/lessons` | 所有者/admin | レッスン追加(末尾に自動採番) |
| PUT | `/lessons/{lesson_id}` | 所有者/admin | レッスン更新 |
| DELETE | `/lessons/{lesson_id}` | 所有者/admin | レッスン削除 |
| PUT | `/courses/{course_id}/lessons/reorder` | 所有者/admin | `lesson_ids`の順序通りに`order`を再採番。コース内レッスンIDの集合と一致しない場合400 |

### 1.3 バリデーション

- `CourseCreate`/`CourseUpdate`: `is_free=false`の場合`price >= 100`を要求、`is_free=true`の場合`price`を強制的に0にする
- `LessonCreate`/`LessonUpdate`: 動画レッスンは`youtube_url`が`https://www.youtube.com/embed/`形式であることを`_to_embed_url()`で検証

## 3. ②Stripeコース購入フロー

既存の[payments.py](../backend/app/routers/payments.py)のOrderベースCheckout Sessionフローとは別に、
詳細設計書Section 2.1の仕様通り **Stripe PaymentIntent方式** でコース購入を実装した。

### 3.1 `POST /payments/checkout`

- `course_id`を受け取り、コース存在チェック(404)・`is_free`コースの購入禁止(400)・
  既存の`status='succeeded'`購入レコードがある場合は**409**を返す(重複購入防止)
- `STRIPE_SECRET_KEY`未設定時は503「決済機能は現在準備中です」(既存パターンに合わせた)
- `stripe.PaymentIntent.create(amount=course.price, currency="jpy", idempotency_key=f"{user_id}_{course_id}_{timestamp}")`
  でPaymentIntentを発行し、`Purchase(status="pending")`を作成。レスポンスは`client_secret`等

### 3.2 Webhook処理

既存の`/payments/webhook`ディスパッチャに以下を追加し、Order用Checkout Sessionフローと共存させた:

- `payment_intent.succeeded` → `_handle_course_payment_succeeded()`: `stripe_payment_intent_id`一致する
  `Purchase`を`status='succeeded'`に更新し、コース内全レッスンに対し`LessonProgress(is_completed=False)`を
  (未存在のもののみ)挿入
- `payment_intent.payment_failed` → `_handle_course_payment_failed()`: 一致する`pending`購入を`status='failed'`に更新

いずれもコース購入で作成された`Purchase`が見つからない場合は無処理(no-op)で正常終了するため、
Order用Checkout Sessionで生成されたPaymentIntentのWebhookイベントとも安全に共存する。

## 5. 動作確認

- バックエンド: 変更した全Pythonファイルに対し`ast.parse`による構文チェックを実施、エラーなし
- フロントエンド: `npx tsc --noEmit`で型チェックを実施、エラーなし
- Dockerで実起動し、以下をcurlで確認:
  - コース作成→レッスン追加(`is_preview=true`/`false`混在)→公開(`status='published'`)
  - `GET /courses/{id}`: admin/匿名いずれの視点でも、未購入時は`is_preview=false`のレッスンの
    `body`/`youtube_url`が正しく`null`化されることを確認
  - `PUT /lessons/{id}`(タイトル更新)、`PUT /courses/{id}/lessons/reorder`(順序入れ替え)、
    `POST /courses/{id}/lessons`→`DELETE /lessons/{id}`(追加→削除)が想定通り動作することを確認
  - コース公開時に、その講師をお気に入り登録しているユーザー宛に`notifications`テーブルへ
    `type='new_course'`の通知が生成されることをDB直接参照で確認
  - 講師一覧/詳細・お気に入り追加/一覧取得が正常応答することを確認
  - `POST /payments/checkout`: `STRIPE_SECRET_KEY`未設定(本開発環境)で503「決済機能は現在準備中です」
    を返すことを確認(想定通りの挙動)
  - Webhook処理(`_handle_course_payment_succeeded`)のロジックを、`Purchase`レコードを直接作成した上で
    関数を呼び出す形で検証: `status`が`succeeded`に更新され、コース内の全レッスン分の`lesson_progress`
    (`is_completed=False`)が正しく挿入されることを確認(検証用データは確認後に削除)
  - フロントエンド新規ルート`/instructors`・`/instructors/1`・`/courses/1`・`/purchase-complete?course_id=1`
    がいずれも200を返すことを確認

### 起動時に発見・修正した不具合

`@stripe/stripe-js`・`@stripe/react-stripe-js`を`package.json`に追加して`docker compose build frontend`を
実行したが、`frontend`サービスの`node_modules`は匿名volumeとしてマウントされており、**`docker compose up -d
--force-recreate`をしても匿名volumeはコンテナ間で再利用され、新しいイメージのレイヤーが反映されない**ことが
原因で`Module not found: Can't resolve '@stripe/react-stripe-js'`(`/courses/[id]`・`/purchase-complete`が500)
が発生した。`docker compose rm -f frontend`でコンテナを削除した上で、対応する匿名volume(`docker volume ls
-qf dangling=true`で特定)を明示的に削除し、`docker compose up -d frontend`で再作成することで解消した。

## 6. 既知の制約・残課題

- 本開発環境では`STRIPE_SECRET_KEY`/`STRIPE_WEBHOOK_SECRET`/`NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`が
  未設定のため、実際のカード決済→Webhook受信までのEnd-to-Endフローは検証できていない。
  決済成功時のサーバー側処理(`_handle_course_payment_succeeded`)は、`Purchase`レコードを直接作成して
  関数を呼び出す形で個別に検証済み
- `/courses`・`/instructors`のコース・講師作成/編集UI(講師向け管理画面)は未実装。今回はAPIと
  学習者向け閲覧・購入フローのみ対応(修正点.md Step4「AIコンテンツ生成スタジオ」で講師側UIを実装予定)
- お気に入り登録時の即時通知(プッシュ通知等)は未実装。`notifications`テーブルへの行追加のみ対応
  (Step5「リテンション機能」で通知一覧UI・既読管理等を実装予定)
- レッスンの動画URLは`youtube_url`のみ対応(自前ホスティング動画は未対応)
