# ManaVillage ピボット 既存コードベース調査・修正方針

> 関連ドキュメント: ManaVillage 要件定義書 / 基本設計書 / 詳細設計書
> 本ドキュメントは、Oshi English の既存コードベースを調査した上で、ManaVillage(講師マーケットプレイス)へのピボットに向けた流用・修正・新規実装の方針をまとめたもの。

---

## 1. 既存コードのうち流用できる部分

| 設計書の要件 | 既存実装 | 流用方針 |
|---|---|---|
| **TONE_PROFILE** | `Character.tone_profile`(JSON) + `backend/app/core/character_voice.py` | ほぼそのまま転用可能。`render_tone_profile()`系の関数群は要件定義のB-02/C-03に完全対応 |
| **二段階生成エンジン** | `ArticleTemplate`(素材)→`Article`(キャラ適応) | 構造はそのまま使える。`build_dm_reply_system_prompt()`を「口調変換」用に汎用化すればC-02/C-03に対応 |
| **Anthropic API呼び出し** | `backend/app/core/llm.py` | 薄いラッパーなので継続使用可。ストリーミング対応のみ追加実装が必要 |
| **JWT認証・ロール管理** | `backend/app/routers/auth.py`、`security.py` | `is_admin`を`role`(learner/instructor/admin)のenumに拡張するだけで要件A-02に対応 |
| **Stripe決済基盤** | `backend/app/routers/payments.py` | Checkout Session・Webhook検証・領収書発行ロジックはそのまま流用。Stripe Connect部分のみ新規 |
| **画像アップロード** | `Character.image_url`管理処理 | アバター/バナーアップロードAPI(F-アップロード要件)に転用可 |
| **記事status管理(draft/review/published)** | `Article.status` | `courses.status`(draft/published/unpublished)にほぼ同じ概念で転用可 |
| **顧客=学習者の基本構造** | `Customer`モデル | `users`テーブルへリネーム・拡張のベースになる |

## 2. 削除・大幅修正が必要な部分

- **`Character`の所有構造**: 現在は「運営が管理する固定キャラ(雪菜・零など)」を前提にした設計(`is_preset`フラグ)。これを「`instructor_profiles.id`に紐づく、講師が自由に作成するキャラ」に変える必要がある → `instructor_id`FKの追加と権限チェックの全面見直し
- **`Customer`モデルの肥大化**: `character_id`を1人の顧客が1キャラに紐づく前提(intimacy_points, character_memory等のチャット重視設計)になっている。マーケットプレイスでは1学習者が複数講師のコンテンツを購入する想定なので、この1:1的な結合は解消が必要
- **`Article`の「リクエスト→管理者が手動編集→開封」フロー**: 個別カスタマイズ前提のワークフロー(記事リクエスト・クレジット消費による開封)。コース単位での売買モデル(`courses`/`lessons`)とは購入単位が異なるため、購入・閲覧権限ロジックは作り直しが必要
- **クレジット制(`credit_balance`, `CreditTransaction`)**: 要件定義はコンテンツ単体購入(都度課金)がメインモデル。クレジットありきの`unlock_cost`方式は撤廃かPhase2機能として再設計
- **管理者向け前提のUI(adminタブ構成)**: 講師ダッシュボード(SCR-06〜10)は学習者/講師/管理者の三者構造を前提に作り直しが必要。現行の管理画面はほぼ「運営が全キャラを操作する」設計
- **2FA・親密度・ご褒美(Reward)システム**: チャットコンパニオン体験向けの機能で、マーケットプレイスの当面のスコープには不要。Phase 1では一旦凍結/削除候補

## 3. 新規実装が必要な部分

- **`instructor_profiles`テーブルと講師申請・審査フロー**(A-02, A-03, D-04)
- **`courses`/`lessons`テーブルとコース単位の購入・進捗管理**(`purchases`, `lesson_progress`) — 現行の記事単体ロック方式とは別物として新設
- **講師一覧・講師ページ(SCR-02, SCR-03)のフロントエンド**
- **AIコンテンツ生成スタジオのUI(SCR-08、Step1〜5のステッパー)とコンテンツ相談チャット(C-01)** — バックエンドのプロンプト構築部分は流用できるが、UIとAPIエンドポイント(`/studio/*`)は新規
- **ストリーミングレスポンス対応**(現状のLLM呼び出しは同期的、StreamingResponse実装が必要)
- **お気に入り講師(`favorites`)、通知(`notifications`)** — 新規テーブル・API
- **Stripe Connectによる収益分配**(F-03) — Phase 2だが設計は早めに検討
- **著作権フィルター/警告(B-05, M-02)の運営レビューUI**

---

## 4. 開発の着手順序の提案

要件定義のPhase 1(自社運営ローンチ)を前提に、以下の順で進めるのが筋が良い。

1. **DBスキーマ移行**: `users`(role追加)・`instructor_profiles`・`characters`(instructor_id追加)・`courses`・`lessons`・`purchases`を新設し、既存の雪菜・零データを移行スクリプトで`instructor_profiles`+`characters`+`courses`形式に変換
2. **認証のロール拡張**: `is_admin`→`role` enum化。既存JWT・2FAロジックはそのまま使えるので変更コストは低い
3. **コース購入フロー(Stripe)**: 既存`payments.py`のCheckout/Webhookパターンを`courses`購入向けに複製・改修。最小限で「買えて読める」状態を最速で作る
4. **講師ページ・コース詳細の学習者向けフロントエンド**(SCR-01〜04): まずは静的に近い形でも公開できる状態を目指す
5. **AIスタジオ(二段階生成)のAPI化**: 既存`character_voice.py`のプロンプト構築ロジックを`/studio/generate/raw`・`/studio/generate/voiced`として切り出し、ストリーミング対応を追加
6. **AIスタジオのフロントエンドUI**: ステッパー・プレビュー
7. **お気に入り・通知などリテンション機能**: 最後に積む

最初の山は1〜3(DB移行+決済)。ここが固まれば「雪菜・零を自社講師として並べたマーケットプレイスの最小形」が動く状態になり、検証ポイント(有料コンテンツが実際に買われるか)に最速で到達できる。
