## 1. ①AIスタジオ バックエンドAPI

新規ルーター [backend/app/routers/studio.py](../backend/app/routers/studio.py) を追加した。
全エンドポイントに`get_current_instructor_or_admin`を適用し、所有権チェック(`InstructorProfile.user_id`一致、
adminは無条件許可)を行う。

### 1.1 新規モデル

[backend/app/models/content_draft.py](../backend/app/models/content_draft.py): 二段階生成の途中経過(`theme`/
`structure`/`target_level`/`raw_content`/`voiced_content`/`script_content`/`instructor_id`/`character_id`)を
保持する`ContentDraft`を追加した。`Base.metadata.create_all()`により`content_drafts`テーブルが自動作成される
ことをDocker上で確認済み。

### 1.2 プロンプト設計

[backend/app/core/studio_prompts.py](../backend/app/core/studio_prompts.py)に詳細設計書Section 3.2のStep0〜4の
プロンプトをそのまま実装した。Step3(口調変換)・キャラクタープレビューは既存の
[character_voice.py](../backend/app/core/character_voice.py)の`render_tone_profile()`を流用し、
保存済み`tone_profile`からシステムプロンプトを構築する。

### 1.3 ストリーミング実装

[backend/app/core/llm.py](../backend/app/core/llm.py)に`stream_text()`を追加した。anthropic公式SDKは依存に
追加せず、既存の`generate_text()`と同じhttpxベースでAnthropic Messages APIの`stream: true`応答(SSE)を自前で
パースし、`content_block_delta`イベントのテキスト断片を逐次yieldする。`extract_json()`はAIの応答が
Markdownコードフェンス(` ```json `)で囲まれて返ってきた場合にも対応してJSONを抽出する。

### 1.4 エンドポイント一覧

| メソッド | パス | 種別 | 概要 |
|---|---|---|---|
| POST | `/studio/generate/character` | 非ストリーミング | Step0: キャラクターイメージ説明→TONE_PROFILE JSON提案 |
| POST | `/studio/consult` | 非ストリーミング | Step1: テーマ→タイトル案・構成案・対象レベル |
| POST | `/studio/generate/raw` | ストリーミング(SSE) | Step2: テーマ+構成→生の教材素材。`ContentDraft`を新規作成し、レスポンスヘッダー`X-Draft-Id`でdraft_idを返す |
| POST | `/studio/generate/voiced` | ストリーミング(SSE) | Step3: draft_id+character_id→口調変換。`draft.raw_content`+`character.tone_profile`から変換プロンプトを構築 |
| POST | `/studio/generate/script` | ストリーミング(SSE) | Step4(オプション): draft_id+character_id→YouTube台本 |
| GET | `/studio/drafts` | - | 自分(instructor_id一致)の下書き一覧 |
| GET | `/studio/drafts/{id}` | - | 下書き詳細(raw/voiced/script全文) |
| DELETE | `/studio/drafts/{id}` | - | 下書き削除 |

ストリーミング系エンドポイントは、生成完了後に`raw_content`/`voiced_content`/`script_content`を
該当`ContentDraft`に保存する。AI生成エラー時はストリーム内に`[ERROR] {メッセージ}`を出力して終了する
(レスポンスヘッダーは既に200で送信済みのため、エラーをHTTPステータスでは表現できないことによる設計上の制約。
フロントエンド側でエラーマーカーを検出してトースト表示する想定)。

### 1.5 既存`/characters` APIの拡張(講師セルフサービス化)

タスク指示では既存の`/characters` APIをそのまま流用する前提だったが、実際には全エンドポイントが
`get_current_admin`(管理者専用)に固定されており、講師が自分のキャラクターを作成・編集できなかったため、
本Stepで以下の拡張を行った([backend/app/routers/characters.py](../backend/app/routers/characters.py)):

- 認証を`get_current_admin` → `get_current_instructor_or_admin`に変更(一覧・作成・更新・削除・画像アップロード/削除)
- 所有権チェック`_get_owned_character()`を追加。講師は`Character.instructor_id`が自分の`InstructorProfile.id`と
  一致するキャラクターのみ操作可能(adminは無条件許可)
- `GET /characters/`: adminは全件、講師は自分の所有キャラクターのみ返すようにスコープ変更
- `GET /characters/{id}`(新規): キャラクター詳細取得(ダッシュボード編集画面用。従来は一覧と
  `/theme/{id}`(tone_profileを含まない公開用)のみで、tone_profileを含む単体取得エンドポイントが存在しなかった)
- `POST /characters/{id}/preview`(新規・詳細設計書 Section 4.6準拠): 保存済み`tone_profile`を使い、
  入力テキストをキャラクターの口調に変換してリアルタイムプレビューを返す
- `PATCH /characters/{id}`: 講師は`instructor_id`(所有権の付け替え)を変更できないようリクエストから除外

- Step 0: 自分のキャラクターから選択

  口調変換済み本文をレッスンとして登録→コース詳細ページへ遷移

## 4. 既知の制約・残課題

- 本開発環境では`ANTHROPIC_API_KEY`が未設定のため、実際のAI生成結果(キャラクター設定提案・コンテンツ相談・
  素材生成・口調変換・台本生成・プレビュー)はEnd-to-Endで確認できていない。ストリーミングのインフラ部分
  (draft作成→SSE配信→DB保存、エラー時の`[ERROR]`マーカー出力)は実際にAPIキーなしの状態で動作確認済み
- ストリーミングエンドポイントはレスポンスヘッダー送信後にHTTPステータスでエラーを表現できないため、
  AI生成失敗時は常にHTTP 200 + ストリーム内`[ERROR]`マーカーで通知する設計とした。フロントエンドの
  `StreamingText`コンポーネントはこのマーカーをエラーメッセージとして検出する実装にはなっていない
  (現状はストリーム本文としてそのまま表示される)。次回実装時にマーカーをパースしてエラー表示に変換する
  改善が望ましい
- `POST /courses/{id}/lessons`への保存は口調変換済み本文(`voiced_content`)をテキストレッスンとして
  1件登録するのみ。台本生成結果(`script_content`)はコース保存フローに含まれず、現状は画面上での確認のみ
  (YouTube動画レッスンとしての活用は今後の検討課題)
- 複数レッスンに分割した構成案(Step1の`structure`の各セクション)を、そのまま複数レッスンとして
  保存する機能は未実装(現状はStep2〜4の生成結果をまとめて1レッスンとして保存)
- キャラクター画像のAI生成自体(画像生成API連携)は対象外。既存の`POST /characters/{id}/image`
  (手動アップロード)のみ利用可能
