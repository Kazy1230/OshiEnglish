# リクエストワークフロー

現状、顧客が依頼できる以下4種類のリクエストについて、ワークフローをまとめる。

1. 記事作成依頼（文法・トピック）
2. 演習問題リクエスト（TOEIC/IELTS/英検/TOEFL ×リーディング・リスニング・スピーキング・ライティング）
3. ライティング評価（添削リクエスト）
4. スピーキング評価（添削リクエスト）

---

## 記事管理画面の分岐構成（改修後）

①記事作成依頼・②演習問題リクエストへの対応は、記事管理タブの「依頼記事」ボタンから以下のように分岐する。

```
記事管理
└ 依頼記事
   ├ 記事                      … ①記事作成依頼（文法・トピック）
   │  └ 記事作成 / キャラに適応   （教育記事ストックを使った2段階）
   └ 演習問題                   … ②演習問題リクエスト
      ├ リーディング
      │  └ 記事作成 / キャラに適応  （問題本体ストックを使った2段階）
      ├ リスニング
      │  └ 記事作成 / キャラに適応  （問題本体ストック＋音声情報、2段階）
      ├ ライティング
      │  └ 記事作成（のみ・1段階）
      └ スピーキング
         └ 記事作成（のみ・1段階）
```

- 「記事作成」「キャラに適応」の2段階構成（教育記事ストック／問題本体ストック）は①・②のリーディング・リスニングに適用
- ②のライティング・スピーキングは記事作成のみの1段階（ストック化なし）。キャラ口調の反映は③④の添削・評価フローで行われる

---

## ① 記事作成依頼（文法・トピック）

### 顧客側
1. チャット画面の「📋 リクエスト」→ `RequestArticleModal` を開く
2. カテゴリで「文法」を選び、自由記述でトピック（例：「仮定法」）を入力（固定200クレジット）
3. `topic` 文字列（例："文法：仮定法"）を `grammar_topic` として `POST /messages/me` に送信

### 料金
- 依頼時に **50クレジット**（`ARTICLE_REQUEST_FEE`）を即時消費
- 残り **150クレジット** は記事を本棚で「開封」する時に消費（`unlock_cost`）

### バックエンド
- `Message`（`is_request=True`, `request_status="accepted"`, `grammar_topic`, `credit_cost=200`）を作成
- 顧客に未着手の受注（`Order`, `status="in_progress"`）がなければ自動作成

### 管理者側
- 管理画面「チャット」タブ／「受注リスト」タブにリクエストが表示（バッジ付き）
- 受注から「依頼記事」ボタンを押すと、記事管理タブ内でさらに**「記事作成（第1段階）」「キャラに適応（第2段階）」の2つの選択分岐**が表示され、それぞれ専用の作成画面に遷移する（詳細は下記「改修後フロー」参照）
- 最終的にキャラ口調を反映した記事を `article_type: "request"`、`request_message_id` 等をセットして `status: "published"` で公開
- `unlock_cost` は `credit_cost(200) − 50 = 150` が自動計算される

### 顧客への配信
- 公開時に自動DM「📚 新しい記事が届いています！」が送信され、リクエストの `request_status` は `completed` に
- 本棚では🔒ロック表示（未開封）。開封時に150クレジット消費して全文表示

### 改修後フロー（教育記事の2段階生成）

記事の生成を「キャラ要素を含まない教育記事」と「キャラ口調への適応」の2段階に分離する。

#### 新設テーブル：教育記事ストック（例: `ArticleTemplate`）

| フィールド | 内容 |
|---|---|
| `topic` | トピック（例：「仮定法」「TOEIC Part5」） |
| `difficulty` | 難易度タグ（`easy` / `medium` / `hard`、管理者が手動指定） |
| `content` | キャラ要素を含まない、わかりやすさ重視の英語解説記事本文（例文・Tips含む） |
| `created_at` | 作成日時 |

第1段階で作られた記事はこのテーブルにストックされ、後から別キャラ・別顧客の同トピック依頼が来た際に第2段階の入力として再利用できる（ポリモーフィズム的な使い回し）。

#### 管理画面：依頼記事への対応フロー

依頼が来たら、記事管理タブの「依頼記事」ボタン →「記事」を選ぶと以下の2分岐に進む（分岐構成は上記参照）。

1. **「記事作成」（第1段階）画面**
   - まず教育記事ストックを一覧から目視で検索し、同じトピック・適切な`difficulty`の記事が既にあるか確認する
   - 既存ストックがあれば、この画面はスキップして②へ
   - なければ、キャラ要素を含まない・わかりやすさ重視の英語解説記事を生成するLLMプロンプトをコピー → 結果を貼り付けて`difficulty`を手動指定 → 教育記事ストックに保存

2. **「キャラに適応」（第2段階）画面**
   - 教育記事ストック一覧から、トピックに合う記事を目視で選択
   - 選んだストック記事の内容＋依頼してきたキャラのトーン・性格＋顧客の親密度を反映してリライトするLLMプロンプトをコピー → 結果を貼り付け
   - 完成した記事を `article_type: "request"` として `status: "published"` で公開（既存フローと同じ）

- LLM呼び出しはAPI連携せず、従来通り「プロンプトをコピー→外部LLMに貼る→結果を貼り付け」を**第1段階・第2段階それぞれで1回ずつ（計2回）**行う

## ② 演習問題リクエスト（TOEIC/IELTS/英検/TOEFL ×リーディング・リスニング・スピーキング・ライティング）

①と同じ入口・同じ課金構造（専用モーダルはなく `RequestArticleModal` の延長）

### 顧客側
1. `RequestArticleModal` でTOEIC/IELTS/英検/TOEFLを選び、Part（例「TOEIC：Part5」「英検2級：リーディング」）を選択（200または400クレジット）
2. `grammar_topic` に試験・パート情報をエンコードして `POST /messages/me`

### 料金
- 50クレジット即時消費＋本棚開封時に150または350クレジット（`unlock_cost`）

### 管理者側
- 同様にチャット/受注リストから記事管理タブへ → ただし `article_type: "exercise"` を選択
- `exercise_format`：
  - **リーディング・リスニング** → `"multiple_choice"`（採点自動）
  - **スピーキング・ライティング** → `"written_response"`（後で③④の添削フローへ繋がる）
- `exercise_category`（例「TOEIC Part5」「英検2級 ライティング」）と `exercise_data`（JSON）を設定
- 「📋 演習問題作成用LLMプロンプトをコピー」→ 試験形式に正確な問題（4〜6問）をJSON形式で生成
  - リスニング系は `listening_script`（台本）もLLMが生成 → 管理者がTTSで音声化し、本文中の任意位置に音声を配置してアップロード
- 公開（`status: "published"`）すると `unlock_cost` 自動計算、自動DM「📚 新しい演習問題が届いています！」

### 顧客側（解答）
- 多肢選択式 → `POST /articles/{id}/check-answer` / `submit-mc`（自動採点）
- 記述式（ライティング・スピーキング演習）→ `POST /articles/{id}/submit-written`（この提出が③④の評価対象になり得る）

---

### 改修後フロー（リーディング）

リーディング（`exercise_format: "multiple_choice"`）について、①と同様に「問題本体（キャラ非依存）」と「キャラ適応」を分離する。

#### 新設テーブル①：問題本体ストック（例: `ExerciseTemplate`）

| フィールド | 内容 |
|---|---|
| `exercise_category` | 試験・パート名（例：「TOEIC Part5」「英検2級 リーディング」） |
| `difficulty` | 難易度タグ（`easy` / `medium` / `hard`、管理者が手動指定） |
| `exercise_data`（問題本体のみ） | `questions: [{ prompt, choices[], correct_index }]`（キャラ要素・解説文・score_commentsは含まない） |
| `created_at` | 作成日時 |

#### 新設テーブル②：採点結果記録（例: `ExerciseSubmission`）

| フィールド | 内容 |
|---|---|
| `article_id` | 対象の演習記事 |
| `customer_id` | 顧客 |
| `attempt_number` | 挑戦回数（記事全体は再挑戦可能。挑戦ごとに新しいレコード群を作成） |
| `question_index` | 設問番号 |
| `chosen_index` | 顧客が選んだ選択肢 |
| `is_correct` | 正誤 |
| `time_taken` | その設問の解答にかかった時間 |
| `customer_proficiency_snapshot` | 解答時点の顧客の習熟度（レベル・親密度など）のスナップショット |
| `answered_at` | 解答（ロック）日時 |

→ 将来的なAI活用（傾向分析・適応学習の教材選定など）を想定したログ。

#### 管理画面：演習問題への対応フロー

依頼が来たら、記事管理タブの「依頼記事」ボタン →「演習問題」→「リーディング」を選ぶと以下の2分岐に進む（分岐構成は上記参照）。

1. **「問題作成」（第1段階）画面**
   - `exercise_category`＋`difficulty`で問題本体ストックを一覧から目視で検索
   - 既存ストックがあれば、この画面はスキップして②へ
   - なければ、キャラ要素を含まない問題本体（パッセージ・設問・選択肢・正解index）を生成するLLMプロンプトをコピー → 結果を貼り付け → `difficulty`を手動指定 → 問題本体ストックに保存
   - **JSON検証・整形UIは廃止**。送信時にバックエンドが裏で検証し、不正な場合のみトーストでエラー表示

2. **「キャラに適応」（第2段階）画面**
   - 問題本体ストック一覧から、カテゴリ・難易度に合う問題を目視で選択
   - 選んだ問題本体＋依頼してきたキャラのトーン・性格＋顧客の親密度を反映し、**`explanation_correct`/`explanation_incorrect`/`score_comments`のみ**を生成するLLMプロンプトをコピー → 結果を貼り付け
   - ストックの問題本体（questions/choices/correct_index）と、生成された解説・score_commentsを統合して最終`exercise_data`を作成
   - `article_type: "exercise"`、`status: "published"`で公開（既存フローと同じ。`unlock_cost`自動計算・自動DM・本棚配信）

#### 顧客側：解答フロー（変更）

- 各設問は選択した瞬間に**正誤を表示し、以後その設問は選択不可（ロック）**になる
  - ロック時に`ExerciseSubmission`へ即時記録（`chosen_index`/`is_correct`/`time_taken`/`customer_proficiency_snapshot`/`answered_at`）
- 全設問ロック後、`submit-mc`相当でスコア・`character_comment`を表示
- **記事全体としては再挑戦可能**（`attempt_number`を増やして新しい解答セットとして記録。過去の挑戦履歴も保持）

---

### 改修後フロー（リスニング）

リスニングはリーディングと同じ`exercise_format: "multiple_choice"`の枠組みであり、**専用のフォーマットや判定フラグを持たない**（`audio_url`/`listening_script`/`[[audio:...]]`の有無で動的に「リスニング」と判定される）。
そのため、リーディングで新設した「問題本体ストック」「採点結果記録」「2画面分岐」「設問ロック＋再挑戦」の仕組みを**そのまま流用**し、問題本体（`ExerciseTemplate.exercise_data`）に音声関連の要素を含める形で対応する。

#### 問題本体ストックへの追加要素
- `listening_script`（台本全文）
- `audio_url` / `questions[].audio_url`（共通音声・設問別音声）
- `instructions`・`questions[].prompt`内の`[[audio:URL]]`（複数パートの音声を任意位置に挿入するプレースホルダー）

これらはキャラ要素を含まないため、リーディングの「問題本体」と同じ第1段階（問題作成）でまとめて作成・ストックする。

#### 第1段階「問題作成」画面（リスニング追加分）
- LLMプロンプトの出力に応じて`listening_script`を取得 → TTSで音声化 →「📋演習問題作成用LLMプロンプトをコピー」と同じ画面内の音声アップロード機能（`POST /admin/exercise-audio`）でアップロード
- 返ってきた`[[audio:/static/exercise_audio/xxxx.mp3]]`を、LLM出力JSON内のプレースホルダー位置または`audio_url`/`questions[].audio_url`に貼り付け
- `difficulty`を指定して問題本体ストック（音声情報含む）に保存

#### 第2段階「キャラに適応」画面
- リーディングと同じ。ストックから問題（音声情報含む）を選択し、`explanation_correct`/`explanation_incorrect`/`score_comments`のみをキャラ口調・親密度で生成 → 統合して`exercise_data`を完成させ公開

#### 🎧リスニング判定・解答・採点
- `hasListeningAudio(exercise_data)`による🎧バッジ判定は変更なし（最終的な`exercise_data`に音声情報が含まれていれば自動的にリスニング扱い）
- 解答ロック・`ExerciseSubmission`記録・`attempt_number`による再挑戦も、リーディングと完全に同一の仕組みを使う

---

### 改修後フロー（ライティング・スピーキング）

ライティング・スピーキング（`exercise_format: "written_response"`）は、リーディング・リスニングとは異なり**「記事作成」の1段階のみ**。
問題本体ストック（`ExerciseTemplate`）・`difficulty`タグ・「キャラに適応」段階は存在しない（「お題」自体がキャラ非依存であり、キャラの口調が反映されるのは③④の添削フィードバック側のため）。

依頼が来たら、記事管理タブの「依頼記事」ボタン →「演習問題」→「ライティング」または「スピーキング」を選ぶと、それぞれ「記事作成」画面のみに進む（分岐構成は上記参照）。

#### ライティング：「記事作成」画面
- 既存の`buildExercisePrompt`（`exercise_format: "written_response"`）でお題（英作文のテーマ・指示文）を生成するLLMプロンプトをコピー → 結果を貼り付け
- `exercise_data`には「お題」と `skill: "writing"` を設定（選択肢・正解index・解説・score_commentsは無し）
- `article_type: "exercise"`、`status: "published"`で公開（既存フローと同じ。`unlock_cost`自動計算・自動DM・本棚配信）

#### スピーキング：「記事作成」画面
- ライティングと同様に`buildExercisePrompt`（written_response）でお題（スピーキングのテーマ・指示文）を生成 → 結果を貼り付け
- 生成されたお題テキストをTTSで音声化し、リスニングと同じ音声アップロード機能（`POST /admin/exercise-audio`）でアップロード →「お題音声」として`exercise_data`に設定（`[[audio:URL]]`または`audio_url`）
- `exercise_data`には `skill: "speaking"` を設定
- `article_type: "exercise"`、`status: "published"`で公開（同上）

> 管理画面のフォームには「スキル（ライティング／スピーキング）」のドロップダウンがあり、選んだ値が`exercise_data.skill`として保存される。この値が、顧客の解答提出時にどちらの添削フロー（③ライティング評価／④スピーキング評価）へ連携するかを決定する。

#### 顧客側：解答UI → ③④への自動連携

- **ライティング**（`skill: "writing"`）：お題表示＋テキスト入力欄 → `POST /articles/{id}/submit-written`
  - 提出と同時に、バックエンドが`CorrectionRequest`（`correction_type: "writing"`, `status: "pending"`, `text_content`=解答本文, `source_article_id`=このお題記事のID）を自動作成 → ③ライティング評価のキューに連携される
- **スピーキング**（`skill: "speaking"`）：お題音声の再生UI＋音声/動画ファイルの添付欄＋任意メモ入力欄 → `POST /articles/{id}/submit-written/media`（拡張子・サイズ50MB以下を既存の添削提出と同様に検証し、`backend/app/static/correction_media/` に保存）
  - 提出と同時に`CorrectionRequest`（`correction_type: "speaking"`, `status: "pending"`, `media_url`/`media_type`, `note`, `source_article_id`=このお題記事のID）を自動作成 → ④スピーキング評価のキューに連携される
- いずれも`source_article_id`によって、添削対応時に元の「お題」が管理者側に自動表示される（詳細は③④参照）

## ③④ ライティング評価／スピーキング評価（添削リクエスト）

`CorrectionRequest`には2つの提出経路がある。

- **自由提出**：`CorrectionSubmissionModal` から提出。`source_article_id = NULL`。①の `RequestArticleModal` で「ライティング添削」「スピーキング添削」を選んだ場合もこちらに遷移する
- **演習問題からの提出**：②ライティング・スピーキング演習の「解答提出」から自動作成（`source_article_id` =元のお題記事ID。上記「改修後フロー（ライティング・スピーキング）」参照）

いずれの経路でも管理者側の対応フロー（添削タブ→記事管理タブでのFB作成）は共通。

### 顧客側（自由提出の場合）
- **ライティング**：英文テキストを入力＋任意メモ
- **スピーキング**：音声/動画を録音またはアップロード＋任意メモ
- 画面には「※ 添削は¥1,000です。提出後、キャラクターからのフィードバック記事が本棚に届きます。」と表示

### 送信先API
- テキストのみ → `POST /corrections/me`
- 音声/動画あり → `POST /corrections/me/media`（拡張子・サイズ50MB以下を検証、`backend/app/static/correction_media/` に保存）
- 演習問題からの提出 → `POST /articles/{id}/submit-written`（ライティング）／`POST /articles/{id}/submit-written/media`（スピーキング）
- いずれも`CorrectionRequest`（`status="pending"`）を作成

### 料金
- クレジットは消費されない。代わりに親密度ポイント（`intimacy_points`）が付与される
- 表示されている¥1,000はクレジット制とは別の有料メニュー（注文/決済側）として案内されている想定

### 管理者側：対応の流れ

1. 管理画面「添削」タブ（`CorrectionsTab`）に `correction_type`（✍️ライティング／🎤スピーキング）と提出内容が一覧表示
   - `source_article_id`がある場合、元になった演習記事のお題（`source_article_title`/`source_article_prompt`）も併せて表示される
   - ステータスを `pending → in_progress → completed` で更新可能
2. 「添削記事を作成」→ 記事管理タブに `article_type: "writing_feedback"` または `"speaking_feedback"`、`correction_request_id` が自動入力された状態で開く
   - 元のお題情報があれば「📝 元のお題」として画面上に表示され、FBプロンプトの`originalPrompt`に自動でセットされる
3. **スピーキングの場合：文字起こしの貼り付け**
   - スピーキング提出（音声/動画）は、FBプロンプト生成の前に文字起こしが必要
   - 文字起こしは運営側で**マニュアルで実施**し、FB作成画面のテキストエリアに貼り付けて「💾 文字起こしを保存」で保存（`PATCH /corrections/admin/{id}/transcript` → `CorrectionRequest.transcript`に保存）
   - 保存した文字起こしテキストが、スピーキングFBプロンプトの「生徒の提出内容（`submission`）」として使われる
   - ライティングの場合は顧客のテキスト回答（`text_content`）がそのまま`submission`として使われる（追加作業なし）
4. 「📋 FB作成用LLMプロンプトをコピー」→ 結果を貼り付けて記事として保存・公開

### フィードバック生成
- ライティング：LLMプロンプトが「全体評価→良かった点→改善ポイント→**次に取り組みたい学習ステップ**→模範解答・改善例→覚えておきたいポイント→キャラクター性」の構成で生成
- スピーキング：「全体評価→良かった点→改善ポイント（内容・構成）→改善ポイント（表現・語彙・文法）→**次に取り組みたい学習ステップ**→模範スピーチ例→覚えておきたい表現・フレーズ→キャラクター性」構成＋自然な英語への書き換え例
- 「次に取り組みたい学習ステップ」は、今回の提出内容の強み・弱みを踏まえて、学習者が次に取り組むべき具体的な練習方法を2〜3点提示するセクション（記事を今回のFBだけでなく今後の学習の足がかりにするための新設項目）
- 両者とも演習の元の設問（`originalPrompt`）と顧客の提出内容（`submission`）、キャラのトーン・親密度を反映

### 顧客への配信
- 公開時に自動DM「📚 新しいライティング/スピーキングフィードバックが届いています！」
- `CorrectionRequest.status = "completed"`、`feedback_article_id` をセット
- 本棚にフィードバック記事として表示（通常 `unlock_cost = 0`）
